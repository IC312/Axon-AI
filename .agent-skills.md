## 1. API and Interface Design (`api-and-interface-design`)

### When to Use
- Designing or modifying Express routes (auth, chat, forum, admin endpoints)
- Adding new request/response contracts
- Changing validation logic or error codes
- Any endpoint that other code or frontend depends on

### Why This Matters for Hermes
Your APIs are the contract between backend and frontend. Judges will review:
- Consistency of error codes and messages
- Backward compatibility (don't break existing clients unexpectedly)
- Input validation boundaries (Hyrum's Law: "anything your API allows will be used")

### Workflow Steps

1. **Define the contract first**
   - Write the endpoint signature (method, path, auth required)
   - Write example request/response (success + error cases)
   - Document all required and optional fields
   - Document HTTP status codes (400, 401, 403, 404, 500)

2. **Apply One-Version Rule**
   - If modifying an existing endpoint, consider: will this break existing clients?
   - If yes, use versioning or feature flags (don't just change behavior)

3. **Validate boundaries early**
   - Input size limits (prevent memory bombs)
   - Field type validation (string vs number, array length)
   - Permission checks (who can call this endpoint?)

4. **Verify**
   - Run cURL or Postman tests for success and error paths
   - Ensure error messages don't leak internal details
   - Ensure auth boundaries are enforced

### Example: Review Before Coding

**Bad (prototype approach):**
```javascript
app.post('/api/messages', (req, res) => {
  const msg = req.body;
  // No validation, no error handling
  db.save(msg);
  res.send({ ok: true });
});
```

**Good (production approach, design-first):**
```javascript
/**
 * POST /api/messages
 * Create a message in a conversation
 * 
 * Auth: Required (JWT)
 * Role: Any authenticated user (but only for their own conversations)
 * 
 * Request:
 *   {
 *     "conversationId": "uuid",
 *     "content": "text (1-5000 chars)"
 *   }
 * 
 * Success (201):
 *   { "id": "msgId", "conversationId": "...", "content": "...", "createdAt": "iso8601" }
 * 
 * Errors:
 *   400: Missing conversationId, content too long, invalid format
 *   401: Missing/invalid auth token
 *   403: User not member of conversation
 *   404: Conversation not found
 *   500: Database error (no stack trace to client)
 */
app.post('/api/messages', authMiddleware, async (req, res) => {
  const { conversationId, content } = req.body;
  
  // Validate input boundaries
  if (!conversationId || !content) {
    return res.status(400).json({ error: 'Missing required fields' });
  }
  if (content.length > 5000) {
    return res.status(400).json({ error: 'Content too long (max 5000 chars)' });
  }
  
  // Check permission
  const conv = await db.getConversation(conversationId);
  if (!conv || !conv.members.includes(req.user.id)) {
    return res.status(403).json({ error: 'Not authorized' });
  }
  
  // Safe create
  try {
    const msg = await db.createMessage(conversationId, req.user.id, content);
    res.status(201).json(msg);
  } catch (err) {
    console.error(err); // Log internally
    res.status(500).json({ error: 'Failed to save message' });
  }
});
```

## 2. Security and Hardening (`security-and-hardening`)

### When to Use
**ALWAYS** when:
- Handling user input (chat messages, forum posts, passwords, emails)
- Making database queries (SQL injection risk? input sanitization?)
- Storing secrets (API keys, passwords, tokens)
- Checking permissions (admin routes, ownership checks)
- Integrating external services (Groq, Astra)

### Why This is Critical for Judges

Judges will run **automated security scanners** + manual review. Security is a top evaluation criterion. A single vulnerability (SQL injection, auth bypass, exposed token) can fail your submission.

### Pre-Commit Security Checklist

Before pushing ANY code, verify:

- [ ] **Input Validation:** All user-provided data validated before DB query or response
  - Example: `req.body.content` must be a string, max length 5000
  - Guard against: empty strings, null, undefined, objects that aren't expected

- [ ] **Authentication:** Protected routes use `authMiddleware`, check `req.user` exists
  - Example: Only authenticated users can create messages

- [ ] **Authorization:** Permission checks are explicit, not assumed
  - Example: User can only delete their own posts (check `post.userId === req.user.id`)
  - Example: Admin-only routes use `adminMiddleware`

- [ ] **Password Handling:** Passwords hashed with bcryptjs (cost 10+), never logged, never returned to client
  - Bad: `res.json({ user, passwordHash: user.passwordHash })`
  - Good: Never return any password-related field

- [ ] **Secrets:** No hardcoded API keys, passwords in code or logs
  - Use environment variables only (process.env.GROQ_API_KEY)
  - Use .gitignore to exclude .env files

- [ ] **Error Messages:** No stack traces, internal details, or SQL errors leaked to client
  - Bad: `res.status(500).json({ error: err.message })` ← leaks internal details
  - Good: `res.status(500).json({ error: 'Internal server error' })`

- [ ] **HTTP Headers:** Set security headers where applicable
  - Example: httpOnly cookies for JWT tokens (already done)

- [ ] **Rate Limiting:** Sensitive endpoints (login, AI chat) should have rate limiting
  - Not implemented yet? Document it as a gap.

### Example: Secure vs Insecure Code

**Insecure:**
```javascript
app.post('/api/forum/posts/:id/delete', (req, res) => {
  const post = db.post.findById(req.params.id);
  db.post.delete(post.id); // No ownership check!
  res.json({ ok: true });
});
```
**Risk:** Any user can delete any post.

**Secure:**
```javascript
app.post('/api/forum/posts/:id/delete', authMiddleware, async (req, res) => {
  const post = await db.getPost(req.params.id);
  
  if (!post) {
    return res.status(404).json({ error: 'Post not found' });
  }
  
  // Explicit ownership check
  if (post.authorId !== req.user.id && !req.user.isAdmin) {
    return res.status(403).json({ error: 'Not authorized' });
  }
  
  await db.deletePost(post.id);
  res.json({ ok: true });
});
```

## 3. Test-Driven Development (`test-driven-development`)

### When to Use
- Before fixing a bug (write failing test first, then fix)
- Before adding a feature (write test for behavior you want)
- Before refactoring (ensure tests pass before/after)

### Why This Matters

Judges will check:
- Does the app handle edge cases? (empty input, large input, missing fields)
- Does the app recover gracefully from errors? (invalid token, DB down, Groq API timeout)
- Are features actually working? (manual testing alone is not enough)

### Test Pyramid for Hermes

```
        /\
       /  \  E2E Tests (5%)
      /____\  Real browser, full flow

     /      \  
    / Unit   \ Integration Tests (15%)
   / & Integ \  Database + API combinations

  /          \
 / Unit Tests \ Unit Tests (80%)
/______________\  Functions, modules in isolation
```

### Examples for Hermes

**Unit Test (80%): Password validation**
```javascript
// test/auth.test.js
describe('Password Validation', () => {
  it('should reject empty password', () => {
    expect(validatePassword('')).toBe(false);
  });
  
  it('should reject password < 8 chars', () => {
    expect(validatePassword('short')).toBe(false);
  });
  
  it('should accept valid password', () => {
    expect(validatePassword('SecurePass123')).toBe(true);
  });
});
```

**Integration Test (15%): Login flow**
```javascript
// test/auth.integration.test.js
describe('POST /api/auth/login', () => {
  it('should return 401 for wrong password', async () => {
    const res = await request(app)
      .post('/api/auth/login')
      .send({ email: 'user@test.com', password: 'wrong' });
    expect(res.status).toBe(401);
  });
  
  it('should return JWT for correct credentials', async () => {
    const res = await request(app)
      .post('/api/auth/login')
      .send({ email: 'user@test.com', password: 'CorrectPass123' });
    expect(res.status).toBe(200);
    expect(res.body).toHaveProperty('token');
  });
});
```

**E2E Test (5%): Chat flow**
```javascript
// test/chat.e2e.test.js
describe('Chat Flow', () => {
  it('should create conversation and send message', async () => {
    const browser = await puppeteer.launch();
    const page = await browser.newPage();
    
    // Navigate, login, create conversation, send message
    // Verify message appears on screen
    
    await browser.close();
  });
});
```

### Rationalizations to Reject

**Excuse:** "I'll add tests later"
**Rebuttal:** Tests catch bugs before production. In a competition with strict judges, "later" means bugs stay in. Write tests as you code.

**Excuse:** "This is too simple to test"
**Rebuttal:** Simple code has simple tests. A 3-line validation function takes 5 lines to test. Worth it.

**Excuse:** "We don't have time to run tests"
**Rebuttal:** You have even less time to debug production failures. Automated tests save time.

## 4. Code Review and Quality (`code-review-and-quality`)

### When to Use
Before merging ANY code change (even if you're the only developer):
- Self-review your own code first
- Check for the 5 axes below
- Look for anti-patterns

### Five-Axis Code Review (Judges Use This)

**1. Correctness**
- Does the code do what it's supposed to?
- Are there off-by-one errors, null checks, edge cases?
- Test: Can you manually trace through a failing scenario?

**2. Clarity**
- Can another engineer read and understand this in 5 minutes?
- Are variable names descriptive? Comments explaining why (not what)?
- Test: Cover the code and explain what it does.

**3. Maintainability**
- Is there duplication? (copy-paste code)
- Is the function too long? (>50 lines is suspicious)
- Test: If you need to add a feature here in 6 months, is it easy?

**4. Performance**
- Are there unnecessary database queries? (N+1 problem)
- Are there blocking operations in loops?
- Test: Does it handle 1000 users? 10,000 messages?

**5. Security**
- Is input validated?
- Are permissions checked?
- Can this code be exploited?
- Test: Try to break it.

### Change Sizing Rule

Aim for changes **~100 lines max** per PR/commit.
- Easier to review
- Easier to revert if broken
- Easier to understand in git history

### Self-Review Checklist

Before sharing code:

```
[ ] Correctness: Does it pass my mental model?
[ ] Clarity: Variable names clear? Comments where needed?
[ ] Maintainability: Any copy-paste? Functions too long?
[ ] Performance: Any N+1 queries? Unnecessary loops?
[ ] Security: Input validated? Permissions checked? Secrets safe?
[ ] Tests: Do I have test coverage for this?
[ ] Diff: Is the change focused or does it refactor unrelated code?
[ ] Compatibility: Does this break existing clients?
```

### Example: Before vs After Review

**Before (not ready):**
```javascript
app.get('/api/messages/:cid', (req, res) => {
  let msgs = [];
  const conversations = db.collection('conversations');
  conversations.find().forEach(c => {
    if (c.id === req.params.cid) {
      const messages = db.collection('messages');
      messages.find().forEach(m => {
        if (m.conversationId === c.id) msgs.push(m);
      });
    }
  });
  res.json(msgs);
});
```
**Issues:**
- N+1 queries (scanning all conversations, then all messages)
- No auth check (anyone can read any conversation?)
- Inefficient loops (should use single query with filter)

**After (ready for review):**
```javascript
app.get('/api/messages/:conversationId', authMiddleware, async (req, res) => {
  const { conversationId } = req.params;
  
  // Verify permission: user must be member of conversation
  const conversation = await db.getConversation(conversationId);
  if (!conversation || !conversation.members.includes(req.user.id)) {
    return res.status(403).json({ error: 'Not authorized' });
  }
  
  // Single optimized query with filter
  const messages = await db.getMessagesByConversation(conversationId);
  res.json(messages);
});
```
**Improvements:**
- Single query instead of N+1
- Auth check in place
- Clear intent
- Easier to maintain

## 5. Debugging and Error Recovery (`debugging-and-error-recovery`)

### When to Use
- Tests are failing, code behaves unexpectedly
- Production error reported by judge or user
- Need to fix a bug quickly and correctly

### Five-Step Triage (When Things Break)

1. **Reproduce** - Get the exact failing scenario
   - What input causes the failure?
   - What's the error message?
   - Can you reproduce it consistently?

2. **Localize** - Find the exact line/function where it breaks
   - Add `console.log()` or debugger statements
   - Narrow down the suspect code
   - Example: Is it the API call or the DB query?

3. **Reduce** - Simplify the failing case to minimal reproduction
   - Remove unrelated code
   - Focus on the core issue
   - Example: Instead of full chat flow, test just message creation

4. **Fix** - Make the minimal fix to resolve the issue
   - Don't refactor the whole function
   - Don't fix multiple bugs in one commit
   - Target fix only

5. **Guard** - Add a test or check to prevent regression
   - Unit test for this specific bug
   - Add validation to catch it earlier next time
   - Example: If null pointer bug, add null check

### Stop-the-Line Rule

If a bug is critical (security, crash, data loss), fix it immediately before other work.

### Example: Debugging a Failing Test

**Failing test:**
```javascript
it('should create message', async () => {
  const msg = await createMessage(convId, 'Hello');
  expect(msg.id).toBeDefined(); // ❌ FAILS
});
```

**Step 1: Reproduce**
- Run the test in isolation
- Check: Does it fail every time? Yes.
- Error: `msg is undefined`

**Step 2: Localize**
- Add logs in `createMessage()`
```javascript
const createMessage = async (convId, content) => {
  console.log('Creating message for:', convId);
  const result = await db.create('messages', { convId, content });
  console.log('DB result:', result); // ← Where does it break?
  return result;
};
```
- Find: `result` is null from DB

**Step 3: Reduce**
- Test DB directly: `await db.create('messages', { convId: 'test-id', content: 'hi' })`
- Find: DB expects `conversationId` not `convId` (field name mismatch)

**Step 4: Fix**
```javascript
const createMessage = async (convId, content) => {
  return await db.create('messages', { conversationId: convId, content }); // Fixed field name
};
```

**Step 5: Guard**
```javascript
it('should create message with correct field name', async () => {
  const msg = await createMessage(convId, 'Hello');
  expect(msg.id).toBeDefined();
  expect(msg.conversationId).toBe(convId); // ← Prevents regression
});
```

## Drawbacks & Limitations

**Drawback 1: Adds cognitive load**
- More checklists to remember
- Solution: Start with 1-2 skills, build habits over time

**Drawback 2: Slows initial development**
- Takes more time to write spec + test first
- Solution: You'll save time in debugging and fixes; net benefit is positive

**Drawback 3: No automated enforcement**
- Skills are guidelines, not auto-checks
- Solution: Use linters (eslint), formatters (prettier), test runners (jest) to automate what you can

**Drawback 4: Overkill for very simple changes**
- 10-line bug fix doesn't need full Security checklist
- Solution: Use judgment; apply skills proportionally
